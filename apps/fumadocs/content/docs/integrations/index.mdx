## The short version

- GitHub and GitLab both flow through the same workspace settings.
- Provider-native reviewer assignment uses connected identities, not manual usernames typed into a form.
- Webhooks are received, stored, and then processed by background jobs so the request path stays fast.
- Enterprise hosts are supported when the provider has been onboarded in the app.

## GitHub and GitLab

- Reviews are posted as pull request comments and inline findings when the provider supports anchors.
- Reviewer requests use GitHub usernames or GitLab reviewer IDs resolved from connected GitPal accounts.
- Labels come from the repository's own label list, so the labeler never invents new labels.
- Merge requests follow the same review and label workflow on GitLab.

## Webhooks

- GitPal stores a webhook receipt before processing begins.
- Duplicate deliveries are ignored by design.
- Webhook registration is idempotent. If two syncs race and GitHub reports that the hook already exists, GitPal re-lists hooks and adopts the matching hook instead of reporting a failed sync.
- Background jobs handle the actual review, labeler, and lifecycle work, which keeps the webhook response fast and stable.
- If a delivery fails, observability keeps the receipt and the failure reason together.

## Repository sync health

- The repositories page shows the last successful PR reconciliation and the current sync state for each repository.
- Use `Sync now` to queue a full provider-state reconcile. Failed syncs retain a safe error summary and expose a retry action.
- A duplicate-hook validation error does not require deleting a healthy GitHub hook manually. Retry the sync after deploying the current release; GitPal will reconcile the existing hook.

## Provider-native reviewer assignment

- Reviewer auto-assignment uses the repository access list plus the connected provider identity for each candidate.
- The app avoids assigning the automation actor and skips the PR author when it can resolve both identities.
- The provider binding comes from the connected account, not from a user-typed server label.

## Enterprise hosts

- Enterprise GitHub and GitLab hosts are onboarded inside the auth flow.
- GitPal stores the host configuration once and reuses it for OAuth, webhooks, and API calls.
- Keep the host URL stable so webhooks, OAuth callbacks, and API calls keep resolving to the same provider record.

## Integration checklist

- `Connected account`: has access to the target provider and repository.
- `Repository access`: includes the humans who should be considered for reviewers.
- `Webhook secret`: matches the provider configuration.
- `Delivery URL`: is publicly reachable and stable.
- `Provider identity mapping`: lets GitPal resolve login or reviewer ID for each candidate.
